# Eng-ception 구현 우선순위 및 위험 관리

> MVP → Core → Growth → Scale 단계별 우선순위 정의 및 의존성 맵

---

## 1. 우선순위 등급 정의

### P0: MVP (필수, 이 없으면 앱 불가)
- 핵심 기능 구현 필수
- 사용자 학습 플로우 완성
- Phase 0 (로컬 프로토타입) 목표

### P1: Core (1순위, 앱의 검증 가능성)
- 배포 가능한 상태
- 기본 사용자 보유 메커니즘
- Phase 1~2 (앱스토어 출시 + 첫 매출) 목표

### P2: Growth (2순위, 유지/재방문)
- 학습 몰입도 증대
- 데이터 기반 최적화
- 유시험 기능 (OPIc 트랙)
- Phase 2 중반 이후 목표

### P3: Scale (3순위, 장기 성장)
- 고급 기능
- B2B 준비
- Phase 3 이후 목표

---

## 5. 위험 평가 및 대응 (Risk Assessment)

### 높은 위험 (High Risk)

#### 1. 분해 엔진 정확도
**위험:** Claude API가 일관되지 않은 구조 분류를 반환
**영향도:** Critical (앱 신뢰성 추락)
**발생 확률:** 중간 (처음 사용자 데이터에서는 프롬프트 튜닝 필요)

**대응:**
- Phase 0: 50문장 검증 완료 (taxonomy-validation.md 기반)
- Phase 1: 실제 사용자 입력 로깅 + 월 1회 프롬프트 개선
- Phase 2: confidence < 0.6 → 경량 경로 (표현 중심 안내)

**모니터링:**
```
- 월별 분류 정확도 추적
- confidence 분포 모니터링
- 경량 경로 비율 (목표: 10% 이하)
```

---

#### 2. 앱스토어 심사 거부
**위험:** iOS/Android 심사 정책 위반
**영향도:** High (출시 지연)
**발생 확률:** 낮음 (표준 e-learning 정책 준수)

**대응:**
- Phase 1: 심사 가이드라인 사전 검토
- 앱 아이콘/설명/스크린샷 정책 준수
- Apple 로그인 필수 (iOS)

---

#### 3. Firebase 비용 폭발
**위험:** Claude API + Firestore 읽기/쓰기 비용 초과
**영향도:** Medium (월간 비용)
**발생 확률:** 중간 (1,000+ 사용자 시)

**대응:**
- Phase 0: API 호출 최소화 설계 (3단계 파이프라인 Phase 3에서)
- Phase 1: 비용 모니터링 대시보드 설정
- Phase 2: 캐싱 + rate limiting 구현
- Phase 3: API 비용 50% 감소 목표 (파이프라인 최적화)

**비용 추이:**
```
Week 0-5: $0 (개발)
Week 5-8: $50~100/월 (테스터 100명)
Week 9-14: $150~200/월 (사용자 500명)
Week 15+: $200~300/월 (사용자 1,000+명)
```

---

### 중간 위험 (Medium Risk)

#### 4. 사용자 유지율 부족
**위험:** 7일 재방문율 < 20%
**영향도:** High (비즈니스)
**발생 확률:** 중간

**대응:**
- Phase 2: 일일 알림 + 스트릭 시스템
- Phase 2: 7일 재방문율 >= 30% 검증 (TOEIC 트랙 진출 조건)
- 사용자 인터뷰로 이탈 사유 파악

---

#### 5. 음성 인식 정확도
**위험:** 네이티브 음성 인식이 영어/한국어 혼용 입력 처리 실패
**영향도:** Medium (레벨 3 검증)
**발생 확률:** 낮음~중간

**대응:**
- Phase 1: 네이티브 음성인식 테스트 (한국인 발음)
- Phase 2: Fallback (음성 인식 실패 시 텍스트 입력)
- Phase 3: 음성 → 텍스트 → 구조 평가 파이프라인 검증

---

#### 6. 인증 연동 문제
**위험:** 카카오/구글/Apple 로그인 중 하나 이상 실패
**영향도:** High (사용자 차단)
**발생 확률:** 낮음 (프로덕션 테스트 부족)

**대응:**
- Phase 1: 3개 인증 모두 베타테스트
- Phase 1: 인증 로그 상세 기록
- Fallback: 이메일 + 패스워드 (나중 추가 가능)

---

### 낮은 위험 (Low Risk)

#### 7. 디자인 시스템 일관성
**위험:** 화면마다 다른 스타일
**영향도:** Low (사용성)
**발생 확률:** 낮음 (토큰 기반 설계)

**대응:**
- Theme 파일 중앙화 (theme/colors.ts 등)
- 컴포넌트 주기적 감시

---

#### 8. 성능 이슈 (렉, 배터리)
**위험:** 애니메이션, 음성인식으로 인한 성능 저하
**영향도:** Medium (UX)

**대응:**

## 6. 검증 체크포인트 (Go/No-Go Decision)

### Phase 0 → Phase 1 (Week 3 말)
**Go 조건:**
- [ ] 모든 P0 항목 완료
- [ ] 로컬 동작 안정 (3회 연속 세션 완료 가능)

- [ ] Firebase 설정 준비 완료

**No-Go 사유:**
- 핵심 UI 버그 미해결
- 학습 플로우 logic 오류
- 음성인식 연동 실패

---

### Phase 1 → Phase 2 (Week 7 말)
**Go 조건:**
- [ ] iOS + Android 앱스토어 출시
- [ ] 3명 이상 베타 사용자 피드백 수집
- [ ] Firestore 데이터 정상 저장
- [ ] 기본 통계 표시 정상

**No-Go 사유:**
- 앱스토어 심사 거부
- 데이터 손상 발생
- 주요 버그 미해결

---

### Phase 2 → Phase 3 (Week 14 말)
**Go 조건:**
- [ ] 7일 재방문율 >= 30%
- [ ] OPIc 트랙 완주율 >= 40%
- [ ] 인앱 결제 정상 동작
- [ ] TOEIC 시나리오 준비 완료

**No-Go 사유:**
- 사용자 유지율 20% 이하
- OPIc 검증 데이터 부족
- TOEIC 결제 수익 불충분한 경우 → Phase 4로 건너뛰기 가능

**Go 조건:**
- [ ] 레벨 3 (음성) 기능 검증
- [ ] TOEIC 트랙 유저 테스트 완료

**No-Go 사유:**

- B2B 파일럿 거절

## 7. 성공 메트릭 및 목표

### Phase 0 (로컬 프로토타입)
- 목표: "앱이 학습 플로우를 완성하는가?"
- 메트릭:
  - 세션당 완료 시간: 30~40초
  - Empathy 스킵율: < 30%
  - 전환점 퀴즈 정답율: > 70%

### Phase 1 (앱스토어 출시)
- 목표: "사용자가 받아들이는가?"
- 메트릭:
  - 설치: 100+
  - 7일 재방문율: >= 20%
  - 평균 세션당 시간: 2분 이상
  - 앱 레이팅: 4.0+ (20개 이상 리뷰)

### Phase 2 (첫 매출)
- 목표: "결제 전환율이 의미 있는가?"
- 메트릭:
  - MAU (월활): 500+
  - 결제 전환율: >= 3%
  - 월간 매출: 50,000원 이상
  - 7일 재방문율: >= 30%

### Phase 3 (데이터 해자)
- 목표: "2개 시험 트랙이 검증되는가?"
- 메트릭:
  - OPIc 트랙 완주율: >= 40%
  - TOEIC 트랙 시작 사용자: > 100
  - 음성 발화 피드백 정확도: >= 80%
  - 월간 매출: 500,000원 이상

### Phase 4 (B2B 스케일)
- 목표: "B2B 채널이 열리는가?"
- 메트릭:
  - B2B 파일럿: 1개 이상
  - 커스터마이제이션 요청: > 3
  - 다국어 언어: 1개 이상 (일본어 또는 중국어)

## 10. 롤백/피벗 계획

### 만약 7일 재방문율이 20% 이하라면?
1. Week 4: 사용자 5명 인터뷰 (이탈 사유)
2. Week 5: 앞 3개 기능 우선순위 재정렬
3. Week 6: A/B 테스트 (알림, 스트릭, 카테고리)
4. Week 7: 최고 성과 기능에 리소스 집중

### 만약 OPIc 검증이 40% 완주율 미만이라면?
1. TOEIC 트랙 실행 연기
2. OPIc 특화 기능 (발음 피드백, 실전 모드) 추가
3. OPIc 커뮤니티 파트너십 (온라인 강사 콜라보)
4. 데이터 다시 수집

### 만약 B2B 파일럿 거절이 3개 이상이라면?
1. 1인 개발 → 팀 확장 고려
2. B2C 중심 재편 (모바일 게이미피케이션)
3. Phase 5: 기업 교육 SaaS 아닌 개인용 앱 최적화
